# AnimeTrack 前端视觉系统与像素插画规范

> 文档状态：基于 2026-07-16 当前工作区代码的静态审查。  
> 本文只定义视觉与后续接入原则，不包含页面改造或占位图片。

## 1. 当前视觉系统总结

### 1.1 产品视觉定位

AnimeTrack 当前不是传统“动漫资讯站”，而是一个以个人数据为中心的动画档案与观看分析工具。视觉上同时具备三种特征：

1. **私人档案馆**：衬线展示字体、大圆角面板、暖纸主题、时间轴与季度簿等命名，强调收藏和回顾。
2. **数据仪表盘**：统计卡、折线图、柱状图、热力图、和弦图、进度条与表格构成高密度信息层。
3. **克制的二次元气质**：动画封面是主要内容图像，界面本身没有大量角色装饰，避免了常见动漫站的花哨和拥挤。

因此，后续像素插画应服务于“私人动画档案馆”的情绪表达，而不是把网站改造成游戏 UI 或角色主题站。

### 1.2 页面结构与布局

- 全局采用顶部固定导航，桌面端为居中横向导航，移动端折叠菜单。
- 主内容以宽屏数据布局为主：
  - `PageContainer` 宽版最大 `1660px`，内容版最大 `1440px`。
  - 页面水平内边距为移动端 `16px`、中屏 `24px`、超宽屏 `32–40px`。
  - 首页和番剧列表在桌面端使用约 `8/4` 主侧栏比例。
  - 时间轴在超宽屏使用 `2/1` 内容与汇总栏比例。
- 页面头部逐步统一到 `PageHero`：
  - 大圆角约 `36px`；
  - 标题使用 Noto Serif SC；
  - 描述、操作按钮和统计卡可在桌面横排、移动端纵向堆叠。
- 内容面板主要由 `Panel`、`glass-panel`、`surface-card`、`surface-card-muted` 构成，圆角集中在 `22–36px`。
- 番剧卡片：
  - 网格封面为 `3:4`；
  - 详情封面为 `2:3`；
  - 列表缩略图、时间轴缩略图和首页横向卡片会对同一封面进行不同裁切。

### 1.3 导航

- 桌面导航清楚、层级浅，主馆区与分析馆在视觉上未显式分组，而是全部平铺。
- 当前活动项以 `tag-bg` 底色和主文本色区分，状态克制但识别度足够。
- 移动端菜单为顶部下拉，链接触控区约 `44px` 以上，基本符合移动端要求。
- 主题切换目前使用“循环到下一主题”的单按钮模式，操作轻量，但用户无法在点击前直接比较四套主题。
- 登录与注册页跳过主导航；管理入口仅对管理员显示。

### 1.4 卡片与信息层级

- 一级容器：`glass-panel-strong`，用于 Hero、重点模块和弹窗。
- 二级容器：`glass-panel` / `surface-card`，用于常规内容区。
- 三级容器：`surface-card-muted`，用于卡片内部列表、统计块、时间轴条目。
- 控件表面：`surface-pill` / `surface-input`。
- 信息层级主要依靠：
  - 背景层级；
  - 边框明度；
  - 大圆角；
  - 主、副、弱三级文字；
  - 主题强调色和语义色；
  - 少量阴影，而不是强烈玻璃模糊。

目前视觉系统的优势是层级稳定、主题变量覆盖较广；风险是大圆角面板数量较多，若再加入大面积插画，容易造成每个区块都在争夺注意力。

### 1.5 按钮与交互

- 主操作使用实色主题强调按钮 `theme-accent-button`。
- 次操作使用 `surface-pill` 或各语义色软底按钮。
- 危险操作统一使用 danger 语义色。
- 筛选项使用 `aria-pressed` 与主题软底，逻辑清楚。
- hover 常见轻微提亮、边框变色、上移或放大。
- 全局已处理 `prefers-reduced-motion: reduce`，会关闭动画和过渡。

建议后续保持“每个操作组只有一个实色主按钮”。像素插画不得承担按钮功能，也不应成为唯一的状态识别方式。

### 1.6 响应式设计

- 整体断点依赖 Tailwind 默认的 `sm/md/lg/xl/2xl`。
- 页面头部、统计卡、主侧栏和图表会逐级折叠，结构总体可靠。
- 网格列表为：
  - 移动端 2 列；
  - `md` 3 列；
  - `lg` 4 列；
  - `xl` 5 列；
  - `2xl` 6 列。
- 表格在小屏使用横向滚动；热力图也允许横向滚动。
- 详情页在 `xl` 以下改为封面与正文纵向排列。

插画在移动端必须缩小或隐藏装饰层，不能使 Hero 重新变成左右双栏，也不能增加横向滚动。

## 2. 品牌关键词

核心关键词：

- 私人收藏
- 动画档案
- 观看记忆
- 安静沉浸
- 数据清晰
- 温柔克制
- 夜间书房
- 像素微光

辅助关键词：

- 旧电视
- 录像带
- 书签
- 放映室
- 星光
- 进度刻度
- 时间胶片
- 档案标签

应避免的方向：

- 高饱和霓虹赛博朋克
- 大面积粉色萌系装饰
- 游戏 HUD 式密集描边

- 每个卡片都添加像素边框
- 与真实作品角色高度相似的设计
- 用插画代替必要的数据、错误信息或操作按钮

### 3.3 颜色使用原则

- 插画不可只适配森绿主题。
- 插画主体建议使用独立、稳定的小色板，主题色只作为 10–20% 的点缀。
- 危险状态必须保留 danger 红色语义，不因主题切换而改成主色。
- 像素插画背景优先透明；背景氛围由网页容器和 CSS 变量提供。
- 正文与按钮仍使用现有 CSS 变量，不从插画中取色覆盖。
- 插画中连续使用的颜色建议控制在 12–20 色；单个小图标控制在 6–10 色。

## 4. 字体、形状与空间

- 正文字体：Noto Sans SC，适合数据与中文正文。
- 展示字体：Noto Serif SC，用于页面标题和状态标题，强化档案与书页气质。
- 数字字体：JetBrains Mono，用于统计、日期、集数和技术信息。
- 基础 HTML 字号为浏览器默认值的 `87.5%`，即约 `14px`。
- 大圆角是现有识别特征：
  - 小控件 `8–16px`；
  - 内层卡片 `18–24px`；
  - 常规面板 `28–32px`；
  - Hero `36px`。
- 插画本身应保持硬边像素轮廓，但外层容器继续使用现有圆角，不新增“像素锯齿边框”。

## 5. 当前状态体验审查

### 5.1 Loading

| 场景 | 当前实现 | 评价 |
| --- | --- | --- |
| 全局路由变化 | 顶部 2px 滑动条，固定约 150ms | 只能表示路由已变化，不代表真实请求完成 |
| 首页模块 | `LazyRender` + `PanelSkeleton` | 方向正确，保持布局稳定 |
| 番剧网格/列表 | 专用 `AnimeGridSkeleton` / `AnimeListSkeleton` | 结构匹配度较好 |
| 时间轴页面 | Hero、图表、列表使用多块 `PanelSkeleton` | 合适，不需要插画 |
| 档期簿 | 3 个大面板 Skeleton | 合适 |
| 图谱馆 | 下半区局部 Skeleton；Hero 和条件图表可能先显示零数据 | 加载边界不完全一致 |
| 管理表格 | 表格行 pulse Skeleton | 合适 |
| 备份列表 | 3 个固定高度 pulse 条 | 合适 |
| 详情页 | 仅一行英文 `Loading details...` | 与其他页面不一致，应该改为详情结构 Skeleton |
| Suspense fallback | 管理、备份、番剧、档期簿使用简单文字 | 可用但视觉系统不统一 |
| 登录提交 | 按钮 spinner + 提示块 | 清楚，不需要插画 |
| 初始化向导 | 状态文案与按钮禁用 | 清楚，不需要插画 |

结论：**Loading 应以 Skeleton 为主，不应使用状态插画。** 用户此时预期内容很快出现，插画会增加跳变并误导为“空状态”。

### 5.2 Empty 与无搜索结果

当前已有共享 `EmptyState`，支持 icon、标题、描述、尺寸和表面样式。使用场景包括：

- 空番剧库；
- 空观看历史；
- 时间轴搜索无匹配；
- 时间轴汇总为空；
- 图谱馆无评分、无开始日期、无首播日期；
- 档期簿无季度数据或筛选无结果；
- 首页无最近活动；
- 备份列表为空；
- 管理表格为空或搜索无匹配。

问题：

- 番剧列表的空状态无法区分“片库真的为空”和“当前搜索/筛选没有结果”，会统一显示“暂无番剧记录”。
- 首页“最近在看作品”使用最多 9 个“最近看得太少啦~”空卡补齐网格，容易让页面显得像未完成占位，不是真正的空状态。
- 管理表格与备份页仍使用独立文本状态，没有统一到 `EmptyState`。
- 局部空状态大量使用 Emoji，风格和 Heroicons、数据面板不完全统一。

### 5.3 Error、404、500 与网络失败

| 场景 | 当前实现 | 缺口 |
| --- | --- | --- |
| React 渲染异常 | 根布局内的 `ErrorBoundary` 显示错误卡、重试和刷新 | 能覆盖客户端子树错误，但不是 Next 路由级错误文件 |
| 404 | 未发现 `app/not-found.tsx` | 当前依赖 Next 默认 404，与品牌视觉不一致 |
| 页面级 500 | 未发现 `app/error.tsx` | 路由错误恢复能力不足 |
| 根级 500 | 未发现 `app/global-error.tsx` | 根布局自身错误没有品牌化兜底 |
| SWR 列表请求失败 | 多数 hook 未读取 `error` | 失败时容易被表现成空数组或旧缓存，用户不知道网络出错 |
| 番剧详情请求失败 | 自动跳回列表 | 错误原因不可见，404 与网络失败被混在一起 |
| 管理表格请求失败 | 未读取 SWR `error` | 可能显示空表格，缺少重试入口 |
| 备份列表请求失败 | Toast 后结束 loading | 页面主体可能显示“暂无备份”，把失败误判为空数据 |
| 登录网络失败 | 表单内显示错误文案 | 合适 |
| 初始化状态失败 | 页面内 danger 提示 | 合适 |
| 写操作失败 | Toast | 对短暂操作合适 |
| 502/503/504 | `fetchJson` 已转换成友好中文提示 | 文案基础已具备，但读取型页面缺少承载这些错误的 UI |
| 浏览器离线 | 无 `navigator.onLine` 或离线状态 | 无明确离线提示 |

结论：错误视觉的优先级高于普通空状态插画。404、500、网络失败是最需要品牌化和明确恢复操作的场景。

## 6. 插画与 Skeleton 的使用边界

### 6.1 适合完整插画

- 404：用户进入了不存在的位置，需要“迷路”主题和返回入口。
- 全页 500 / 全局错误：需要缓和挫败感，同时突出重试。
- 网络失败 / 离线：需要明确区分“没有数据”和“数据没加载到”。
- 首次空片库：这是产品的关键起点，适合用吉祥物引导添加第一部作品。
- 首次空观看历史：可以表达“放映尚未开始”，但优先级低于空片库。

### 6.2 适合小型插画或像素图标

- 搜索无结果；
- 空备份列表；
- 档期簿无首播数据；
- 图谱馆无评分、无开追日期；
- 局部活动流为空。

这些场景应使用 `96×96` 左右的小型透明素材，不应占据整张面板。

### 6.3 应继续使用 Skeleton

- 首页首次加载；
- 番剧列表加载与翻页；
- 详情页加载；
- 时间轴、热力图与图表加载；
- 档期簿加载；
- 图谱计算与图表加载；
- 管理表格与备份列表加载；
- 懒加载模块进入视口前。

Skeleton 应尽量匹配最终结构，避免仅用整块灰色矩形。加载超过约 8 秒时，再叠加简短文案和重试提示，不使用插画替代加载反馈。

### 6.4 不建议使用插画

- 单个字段缺失，如“暂无简介”“未补充声优”“未补充原名”；
- 卡片封面加载失败；
- 表单校验错误；
- Toast；
- 确认弹窗；
- 小型图表没有数据；
- 管理页权限验证；
- 主题切换器和普通导航。

## 7. 像素插画风格规范

### 7.1 风格定义

建议命名为：**Pixel Archive / 像素档案馆**。

视觉语言：

- 16-bit 与现代精细像素画之间的中等颗粒度；
- 清晰的 1–2 个逻辑像素轮廓；
- 低饱和深色主体，局部使用主题色微光；
- 以旧电视、录像带、档案盒、书签、星光、播放符号为道具；
- 角色采用原创简化造型，不复刻现有动画 IP；
- 阴影用色块和少量抖动，不使用模糊画笔；
- 不在插画内部绘制长文本，避免多语言和缩放问题。

### 7.2 画面密度

- 主体占画布 55–72%。
- 保留 12–16% 安全边距。
- 每张状态插画只表达一个主要动作。
- 同屏不出现超过一个完整角色。
- 道具最多 3 类，避免像素噪声。
- 表情应在 `96px` 显示宽度下仍可辨认。

### 7.3 轮廓与阴影

- 外轮廓使用深绿黑、深棕黑或中性炭黑，不使用纯黑硬套所有主题。
- 亮面最多 3 个明度层级，阴影最多 2–3 个层级。
- 禁止柔边投影、镜头光晕和写实景深直接烘焙到透明图内。
- 光晕、柔和底板和主题氛围由 CSS 容器提供。

### 7.4 网页缩放

- 使用整数倍最近邻缩放。
- 逻辑像素画布建议以 `96×96`、`160×120`、`192×128`、`96×128` 为基础。
- 导出时放大 4 倍，确保高 DPI 屏幕清晰。
- CSS 显示尺寸尽量是逻辑尺寸的整数倍或二分之一；避免任意小数缩放。
- 接入时可使用 `image-rendering: pixelated`，但仍应准备足够大的 4x 导出文件。

## 8. 像素尺寸与网页显示尺寸

### 8.1 状态插画

| 类型 | 逻辑画布 | 导出像素 | 桌面显示 | 移动端显示 | 比例 | 背景 |
| --- | --- | --- | --- | --- | --- | --- |
| 全页错误/404 | `160×120` | `640×480` | `280–320×210–240` | `200–240×150–180` | `4:3` | 透明 |
| 全页空片库 | `192×128` | `768×512` | `320–384×213–256` | `224–288×149–192` | `3:2` | 透明 |
| 网络失败 | `160×120` | `640×480` | `256–320×192–240` | `200–240×150–180` | `4:3` | 透明 |
| 局部空状态 | `96×96` | `384×384` | `96–128×96–128` | `80–112×80–112` | `1:1` | 透明 |
| Hero 装饰 | `192×128` | `768×512` | 最大宽 `360–420` | 最大宽 `180–240` 或隐藏 | `3:2` | 透明 |
| 吉祥物半身 | `96×128` | `384×512` | `144–192×192–256` | `96–128×128–171` | `3:4` | 透明 |

### 8.2 现有封面容器

| 场景 | 当前容器 | 典型显示尺寸 | 建议源图 |
| --- | --- | --- | --- |
| 番剧网格卡 | `aspect-[3/4]` | 约 `150–190×200–253` | 至少 `600×800`，推荐 `768×1024` |
| 番剧详情 | `aspect-[2/3]`，桌面宽 `360–390` | 约 `360×540` 至 `390×585` | 至少 `720×1080` |
| 番剧列表 | `56×74` | 固定小缩略图 | 复用主封面 |
| 时间轴条目 | `44×64` | 固定小缩略图 | 复用主封面 |
| 时间轴汇总 | `40×56` | 固定小缩略图 | 复用主封面 |
| 侧栏最近观看 | `40×40` | 方形中心裁切 | 复用主封面，主体应靠中 |
| 首页最近在看 | `aspect-video` | 约 `220–320×124–180` | 复用主封面作背景裁切 |
| 图谱评分卡 | 高 `160px`，宽随列变化 | 横向中心裁切 | 复用主封面作背景裁切 |
| 首页 Hero 背景 | 整个 Hero cover | 宽屏大面积裁切 | 复用封面，仅作低透明背景 |

封面并没有单一安全比例。若继续使用同一张竖版封面，内容主体应尽量位于中央 60% 区域，并接受在 `3:4`、`2:3`、`16:9` 和 `1:1` 中被裁切。状态插画不要复用封面规则，应使用独立透明资产。

## 9. 吉祥物设计建议

### 9.1 角色定位

建议设计一只原创的“档案放映员”吉祥物，暂名 **小迹 / Track-chan**，但正式命名可另行决定。

角色功能不是常驻聊天助手，而是：

- 引导首次使用；
- 陪伴错误恢复；
- 统一状态插画；
- 在少量品牌位置建立记忆点。

### 9.2 外形建议

- 小型非人类或弱人形角色，避免与具体动画角色相似。
- 核心轮廓可由“猫耳式胶片盒 + 围巾/书签尾巴 + 小型放映机包”组成。
- 身体主色使用深中性色，眼睛或胸前指示灯使用主题绿/青点缀。
- 保留一个固定识别物：三角播放徽记、胶片尾巴或星形书签，三选一即可。
- 表情系统至少包含：
  - 平静；
  - 寻找；
  - 困惑；
  - 抱歉；
  - 鼓励；
  - 睡眠/等待。

### 9.3 使用限制

- 不作为所有卡片的水印。
- 不覆盖真实番剧封面。
- 不在数据图表中作为数据点。
- 同一页面最多出现一次完整吉祥物。
- 管理页和高密度表格默认不出现完整角色。

## 11. 深浅主题适配

### 11.1 推荐方案

优先使用“一套透明主体 + 主题化网页底板”：

- 插画文件自身保持透明背景。
- 插画轮廓使用跨主题稳定的深炭色和暖灰色。
- 容器使用现有 `bg-card`、`tag-bg`、`accent-light`、`secondary-light` 提供主题氛围。
- 可在插画后加入 CSS 径向渐变，但不得烘焙进图片。

### 11.2 何时需要双版本

只有在以下情况制作 `-dark` / `-light` 两版：

- 深色轮廓在 Lumen 暖纸主题对比不足；
- 浅色高光在 Noir 或 Verdant 中消失；
- 画面包含大面积半透明烟雾或星点，无法通过外层底板修正。

命名示例：

- `state-404-lost-track-dark.png`
- `state-404-lost-track-light.png`

不建议为四个主题分别制作四套完整插画。主题差异应主要由 CSS 背景、边框、按钮和光晕承担。

### 11.3 对比检查

- 插画关键轮廓对所在容器至少达到约 `3:1` 的非文本视觉对比。
- 标题、说明、按钮仍需满足正常文本对比要求，不能把文字画进图片。
- danger、success、watching 等语义必须由 UI 文案和控件重复表达。

## 12. 图片性能规范

- 状态插画优先使用带透明通道的 WebP；若像素边缘在编码后出现色边，改用 PNG。
- 不使用 JPEG 保存透明像素插画。
- 全页状态插画单张目标：
  - WebP `30–90KB`；
  - PNG 尽量不超过 `180KB`。
- 小型状态图目标不超过 `40KB`。
- Hero 装饰图目标不超过 `120KB`。
- 使用无损或近无损导出，禁止插值缩放。
- 图片文件应放在明确目录，例如：
  - `public/illustrations/states/`
  - `public/illustrations/mascot/`
- 首屏只预加载真正首屏可见的品牌图；错误页和空状态图按页面加载。
- 番剧封面当前多处使用 `unoptimized` 或 CSS background，后续如做性能优化，应统一图片加载策略，但不属于插画接入的前置条件。
- CSS background 图片无法天然获得 `alt`、响应式 `sizes` 与加载占位；状态插画应优先使用语义化 `<Image>` / `<img>`。

## 13. 无障碍规范

- 纯装饰插画：`alt=""`，必要时加 `aria-hidden="true"`。
- 承载状态含义的插画：使用短 alt，例如“吉祥物正在调整无信号电视”，但状态原因和恢复操作仍必须写在正文中。
- 不在图片中嵌入关键文字、错误码、按钮标签或操作说明。
- 404、500、网络失败页面应有清晰的 `<h1>`。
- 错误状态应提供至少一个可执行操作：
  - 重试；
  - 刷新；
  - 返回首页；
  - 返回番剧列表。
- 网络失败与空数据必须使用不同标题与说明。
- Skeleton 建议统一 `aria-hidden="true"`，其父区域使用 `aria-busy="true"`；必要时提供屏幕阅读器专用“正在加载”文本。
- 动画像素效果需服从 `prefers-reduced-motion`，不要让吉祥物持续闪烁或抖动。
- 图片不应抢占键盘焦点。

## 14. 后续组件接入建议

### 14.1 状态模型

建议先把状态明确分成：

```text
loading
success-with-data
success-empty
success-no-results
request-error
offline
not-found
render-error
```

不能再用 `data ?? []` 把请求失败自动折叠成空数据。

### 14.2 共享组件方向

可在后续实现时扩展或新增：

- `EmptyState`
  - 增加 `illustration`、`secondaryAction`、`tone`；
  - 区分 `empty` 与 `no-results`。
- `ErrorState`
  - 支持 `network`、`server`、`not-found`；
  - 统一标题、说明、错误码、重试和返回操作。
- `StateIllustration`
  - 统一图片路径、宽高、主题版本、alt 和移动端尺寸。
- `DetailSkeleton`
  - 匹配详情页 `2:3` 封面和正文双栏。
- `TableStateRow`
  - 管理表格内统一 loading、empty、no-results、error。
- `OfflineBanner`
  - 仅在离线或重连中显示，避免把整个旧数据页面替换掉。

### 14.3 推荐接入顺序

1. 新增 Next 路由级 `not-found`、`error`、`global-error`，接入 P0 插画。
2. 让 `useAnimeData`、`useHistoryData` 和各 SWR 页面暴露并渲染 `error`。
3. 区分番剧列表空库与筛选无结果。
4. 将详情页和文字 Suspense fallback 改为结构化 Skeleton。
5. 统一备份页、管理表格与局部模块的 Empty/Error 状态。
6. 接入首次空片库插画。
7. 根据真实使用频率逐步补充 P1、P2 小型状态图。

### 14.4 接入时的布局原则

- 全页状态内容宽度建议 `max-width: 560px`，插画、标题、说明、按钮纵向排列。
- 插画与标题间距 `20–28px`。
- 标题与说明间距 `8–12px`。
- 操作按钮组距说明 `20–24px`。
- 移动端状态容器左右留白至少 `20px`。
- 局部状态图不超过面板高度的 40%。
- Hero 装饰图放在右侧或背景层，正文区域至少保留 60% 可用宽度。
- 在 `md` 以下优先缩小或隐藏 Hero 装饰，不把正文挤到过窄。

## 15. 当前系统的具体视觉风险

- `theme-dashboard-aura`、`theme-atlas-aura`、`theme-atlas-hero-aura`、`theme-detail-aura` 在组件中被引用，但当前 `globals.css` 中未发现对应定义；这些装饰层目前可能没有实际视觉效果。
- 全站没有品牌化的 Next 404、路由错误和根错误文件。
- 多个 SWR 页面忽略请求错误，存在把网络失败表现为空状态的风险。
- 详情页加载与部分 Suspense fallback 仍是英文或简单文字，与共享 Skeleton 系统不一致。
- 封面在 `3:4`、`2:3`、`16:9` 和 `1:1` 间多次裁切，人物或标题容易被截断。
- 首页用空卡补满“最近在看”网格，视觉上容易被理解成加载占位。
- Emoji 空状态与 Heroicons、像素插画三套图形语言并存；后续应逐步收敛，但不需要一次替换所有局部状态。
- Lumen 是唯一浅色主题，插画对比不能只在三个深色主题中验收。

## 16. 最终视觉方向

AnimeTrack 最适合的不是“满屏像素化”，而是：

> 以现有沉稳的数据档案界面为主体，用少量原创像素插画承接空白、迷路、故障和首次使用时刻。

真实动画封面继续承担内容视觉；Skeleton 负责速度感和布局稳定；像素吉祥物只在具有情绪价值的状态中出现。这样既能强化二次元识别度，也不会破坏当前克制、成熟、偏个人收藏工具的整体气质。
